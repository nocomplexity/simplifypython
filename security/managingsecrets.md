# How To manage Secrets

## The Challenge

Managing secrets securely is a persistent challenge in software development. Any developer with even basic security awareness knows that hardcoding secrets—such as API keys, passwords, or tokens—into source code is inherently risky.

In practice, however, convenience often overrides security. For instance, when making REST calls to GitHub, developers frequently require rapid access to an API key. This often leads to shortcuts that ultimately create significant security weaknesses.

There are **simple, good, perfect, and bad** ways to handle secrets. The difficulty lies in consistently choosing the most robust approach for the task at hand.


## The Threat

The primary threat is exposure. Once secrets are embedded in code, they can easily be leaked through:

* Version control systems (e.g. public repositories)
* Logs or debug outputs
* AI-generated code snippets reused without scrutiny
* Misconfigured deployments

The internet is currently saturated with autogenerated “solutions” (such as those found on sites like Beagle Security), but many of them overlook subtle vulnerabilities.

Even when developers attempt to follow best practices—such as using environment variables—poor implementation can still lead to **weaknesses that are easy to exploit**.


## Vulnerable Code Example

A common insecure pattern looks like this:

```python
API_KEY = "my_super_secret_key"

def call_github():
    headers = {"Authorization": f"Bearer {API_KEY}"}
    # make request
```

Why is this dangerous?

* The secret is visible in plain text.
* It may be committed to a repository accidentally.
* Anyone with access to the codebase has access to the secret.

Even slightly “better” approaches—such as storing secrets in a `.env` file—remain risky if:

* The file is accidentally committed to version control.
* File permissions are too broad.
* Secrets are not regularly rotated.


## Secure Mitigation

Rather than embedding secrets in code, there are far safer alternatives:

**1. Environment Variables**
Use environment files such as `.env` or `.config` and load them securely. A common tool is the [python-dotenv library](https://github.com/theskumar/python-dotenv). However, be aware: this module contains several security weaknesses if not handled with care.



**2. Configuration Files**
Store secrets in external files (JSON, INI, config), but ensure:

* Data is properly encrypted at rest.
* Access permissions are strictly restricted.

**3. Secrets Managers**
Utilise dedicated secret management systems that provide:

* Granular authentication and authorisation.
* Full audit logging.
* Automatic credential rotation.

Popular solutions include:

* AWS Secrets Manager
* Google Cloud Secret Manager
* Azure Key Vault
* **Key Management Systems (KMS):** A KMS helps securely generate, store, and manage cryptographic keys. A well-known enterprise option is HashiCorp Vault.

**4. [Practice Security-By-Design](https://nocomplexity.com/documents/securitybydesign/intro.html)**
Design your architecture and implementation to be secure by default. Ensure you **minimise** the burden of manual secret and key management.


## Discussion

Is using environment variables or `.env` files secure enough?
**Short answer: No.**

While better than hardcoding, it is not foolproof. Weak access controls, poor operational practices, or a lack of auditing can quickly turn these methods into vulnerabilities.

The real solution lies in **[Security-by-Design](https://nocomplexity.com/documents/securitybydesign/intro.html)**:

* **Audit before use:** Evaluate any code before implementation (especially AI-generated snippets).
* **Centralise:** Prefer centralised, managed secret stores over local files.
* **Least Privilege:** Implement least-privilege access across all environments.
* **Monitor:** Enable logging and monitoring of secret usage.
* **Rotate:** Regularly rotate credentials to limit the impact of a potential leak.

The best advice: do not just “hide” secrets—**manage them properly.**
